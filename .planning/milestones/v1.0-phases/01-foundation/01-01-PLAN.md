<task type="auto">
  <name>Task 2: Create shared-nav.js IIFE module</name>
  <files>custard-calendar/docs/shared-nav.js</files>
  <action>
Create `docs/shared-nav.js` as an IIFE following the existing `CustardPlanner` pattern. The module auto-initializes on DOMContentLoaded and injects into `<div id="shared-nav">`.

**Module structure (window.SharedNav IIFE):**

```
var SharedNav = (function() {
  'use strict';
  // ... all code ...
  return { renderNav, showStorePicker, updateStoreIndicator };
})();
```

**1. Nav rendering (renderNav):**
- Find `document.getElementById('shared-nav')`.
- Build nav HTML with current/legacy links (same 11 items currently in all pages): Forecast (index.html), Calendar, Map, Radar, Alerts, Siri, Fronts (forecast-map.html), Quiz, Widget, The Scoop (scoop.html), Group.
- Detect current page via `window.location.pathname` to set `nav-active` class.
- Render store indicator above the nav links.
- Use `CustardPlanner.escapeHtml()` for any dynamic text.

**2. Store indicator (buildStoreIndicator):**
- Read slug via `CustardPlanner.getPrimaryStoreSlug()`.
- If slug exists: fetch store metadata (name, city) from the stores manifest endpoint (`CustardPlanner.WORKER_BASE + '/api/v1/stores'`). Cache the response in sessionStorage key `'custard:store-manifest'` to avoid re-fetching on every page.
- While the manifest is loading (async), show the slug formatted nicely (replace hyphens with spaces, title-case) as a placeholder.
- Once loaded: render compact indicator: `[Store Name, City] [change]`.
- The "change" text must be a tappable `<button>` (not just a link) with class `store-change-btn`.
- Must fit at 375px width. Use CSS text-overflow: ellipsis for long store names.

**3. First-visit geolocation flow (STOR-01, STOR-02):**
- If `CustardPlanner.getPrimaryStoreSlug()` returns null (no saved store):
  - Passively fetch IP geolocation from `http://ip-api.com/json/?fields=lat,lon,city,regionName` (no user prompt).
  - On success: use `CustardPlanner.haversineMiles()` to find nearest store from the manifest.
  - Display first-visit prompt: "Showing flavors for [Store Name] -- change?" with class `first-visit-prompt`.
  - Do NOT call `setPrimaryStoreSlug()` yet -- wait for user to confirm or change.
  - Add a "Looks good" / confirm button that calls `setPrimaryStoreSlug(nearestSlug)` and transitions to the regular indicator.
  - Add the "change" button that opens the store picker.
  - If IP geolocation fails: show the store picker directly (graceful fallback).
- Do NOT trigger `navigator.geolocation.getCurrentPosition()` automatically. Only offer a "Use precise location" button inside the store picker that triggers it on tap.

**4. Store picker (showStorePicker):**
- Opens when user taps "change" on the indicator OR the first-visit prompt.
- Renders a picker overlay/panel with:
  - A text input with placeholder "Search stores..." and type-ahead filtering.
  - A scrollable list of matching stores (name, city, state).
  - A "Use precise location" button that triggers browser Geolocation API on click.
- Filter logic: match store name, city, or state against input text (case-insensitive substring).
- On store selection: call `CustardPlanner.setPrimaryStoreSlug(slug)`, update the indicator, close the picker.
- Store manifest: fetch from `CustardPlanner.WORKER_BASE + '/api/v1/stores'` if not already cached in sessionStorage.
- Picker must be hidden by default (`display: none` or `hidden` attribute). Only shown on user action.

**5. Auto-init:**
- `document.addEventListener('DOMContentLoaded', function() { ... })` finds `#shared-nav` and calls `renderNav()`.
- If `#shared-nav` element is missing, do nothing (graceful no-op for pages that don't use shared nav).

**Critical constraints (from CONTEXT.md locked decisions):**
- ES5-compatible IIFE (no `import/export`, no arrow functions, no `let/const` -- use `var`).
- No browser geolocation permission prompt on page load.
- IP geolocation first, browser geolocation only on explicit user action.
- Store picker hidden by default on all pages.
- During Phase 1, nav shows current/legacy 11 items (Phase 4 switches to 4-item nav).

**Race condition prevention (from RESEARCH.md Pitfall 2):**
- Create a `storeManifestReady` promise that resolves when the store manifest is fetched.
- Geolocation nearest-store lookup must `storeManifestReady.then(...)` before attempting distance calculation.
